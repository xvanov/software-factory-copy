1"""Azure provider bootstrap (covers BOTH Foundry and Azure-OpenAI shapes). 

2 

3Two distinct Azure surfaces are supported. They live in this module together 

4because they share env-remap mechanics and a single idempotent bootstrap 

5entry point; the wire shapes diverge entirely. 

6 

71. **Azure AI Foundry** (``services.ai.azure.com/models``-style endpoints) 

8 — addressed in LiteLLM via the ``azure_ai/`` prefix. Foundry exposes an 

9 OpenAI-compatible chat-completions endpoint at 

10 ``<base>/chat/completions?api-version=<version>``. LiteLLM auto-downgrades 

11 to the Azure-OpenAI deployment URL whenever the model name (e.g. 

12 ``gpt-4.1``) appears in ``litellm.open_ai_chat_completion_models``; we 

13 monkey-patch that detection off so every ``azure_ai/...`` id uses the 

14 simpler OpenAI-compatible path. 

15 

162. **Azure OpenAI / Cognitive Services** (``cognitiveservices.azure.com`` 

17 endpoints with named deployments) — addressed in LiteLLM via the plain 

18 ``azure/`` prefix. Each deployment is reached at 

19 ``<base>/openai/deployments/<deployment>/chat/completions?api-version=X``. 

20 LiteLLM reads ``AZURE_API_BASE`` / ``AZURE_API_KEY`` / ``AZURE_API_VERSION`` 

21 from the environment automatically — we only need to remap the friendlier 

22 ``AZURE_ENDPOINT`` name into ``AZURE_API_BASE``. 

23 

24``ensure_bootstrapped()`` is idempotent and is called at the start of every 

25Azure-bound runner code path. It: 

26 

27 1. Remaps ``AZURE_FOUNDRY_*`` env vars → LiteLLM's ``AZURE_AI_API_*`` names 

28 (Foundry path). 

29 2. Remaps ``AZURE_ENDPOINT`` → ``AZURE_API_BASE`` (Azure-OpenAI path). 

30 ``AZURE_API_KEY`` / ``AZURE_API_VERSION`` already match LiteLLM's 

31 expected names and are left as-is. 

32 3. Sets ``litellm.drop_params = True`` so newer reasoning-class models 

33 (``gpt-5.4`` etc.) that reject the legacy ``max_tokens`` get it 

34 translated to ``max_completion_tokens`` instead of 400-ing. 

35 4. Monkey-patches LiteLLM's Foundry detection. This is a no-op for the 

36 plain ``azure/`` path; it only affects ``azure_ai/`` calls. 

37 

38We never overwrite a value the operator set explicitly. 

39""" 

40

49def _remap_env() -> None: 

50 """Copy operator-friendly env-var names into LiteLLM-expected ones. 

51 

52 Two independent remaps: 

53 

54 * ``AZURE_FOUNDRY_*`` → ``AZURE_AI_API_*`` (Foundry / ``azure_ai/`` path). 

55 * ``AZURE_ENDPOINT`` → ``AZURE_API_BASE`` (Azure-OpenAI / ``azure/`` path). 

56 

57 ``AZURE_API_KEY`` / ``AZURE_API_VERSION`` are already LiteLLM-standard 

58 names and are left untouched. 

59 

60 We never overwrite a destination that is already set explicitly. 

61 """ 

62 pairs = [ 

63 # Foundry (azure_ai/...) path 

64 ("AZURE_FOUNDRY_ENDPOINT", "AZURE_AI_API_BASE"), 

65 ("AZURE_FOUNDRY_API_KEY", "AZURE_AI_API_KEY"), 

66 ("AZURE_FOUNDRY_API_VERSION", "AZURE_AI_API_VERSION"), 

67 # Azure-OpenAI (azure/...) path — operator-friendly alias for AZURE_API_BASE 

68 ("AZURE_ENDPOINT", "AZURE_API_BASE"), 

69 ] 

70 for src, dst in pairs: 

71 src_val = os.environ.get(src) 

72 if src_val and not os.environ.get(dst): 

73 os.environ[dst] = src_val

97def _enable_litellm_drop_params() -> None: 

98 """Make LiteLLM silently drop / translate provider-unsupported params. 

99 

100 ``gpt-5.4`` (and other 2026 reasoning-class models) reject the legacy 

101 ``max_tokens`` parameter and want ``max_completion_tokens`` instead. With 

102 ``drop_params=True``, LiteLLM auto-translates rather than 400-ing, so 

103 callers can keep passing ``max_tokens`` uniformly. 

104 """ 

105 try: 

106 import litellm 

107 

108 litellm.drop_params = True 

109 except Exception: # pragma: no cover 

110 return

113# Estimated per-token costs for ``azure/deepseek-v4-pro``. LiteLLM ships 

114# without a pricing entry for this Azure deployment, so every sandbox dev / 

115# test_implementer run lands a ``cost=0.0`` row in ``state/factory.db.runs`` 

116# even though the model burns 700K+ tokens per call. Without a price, the 

117# chain's spend caps are useless against the heaviest model. 

118# 

119# Verified 2026-07-18 against the Azure retail price API (eastus2, the 

120# ``FW DeepSeek-V4-Pro`` meters — the only published tier; our deployment is 

121# GlobalStandard, whose rate matches or slightly undercuts Data Zone): 

122# input $0.00193 / 1K = $1.93 / 1M 

123# output $0.00383 / 1K = $3.83 / 1M 

124# The prior estimate ($0.50 / $1.50 per 1M) UNDER-counted dev spend ~3.8x on 

125# input and ~2.5x on output — dev/test_implementer bulk runs on this model, 

126# so historical ``runs.cost_usd`` for those rows reads low by roughly that 

127# much. (Cost Management actuals would be the final word but this account 

128# lacks the RBAC role for the billing API; retail list price is the 

129# authoritative published rate.) 

130_DEEPSEEK_V4_PRO_INPUT_PER_TOKEN = 0.00000193 # $1.93 per 1M (Azure retail, eastus2) 

131_DEEPSEEK_V4_PRO_OUTPUT_PER_TOKEN = 0.00000383 # $3.83 per 1M (Azure retail, eastus2) 

132 

133# D003 audit finding (2026-07-19): this registration had NO 

134# ``cache_read_input_token_cost``. LiteLLM's ``generic_cost_per_token`` prices 

135# cache-hit prompt tokens at whatever that key resolves to — and its 

136# ``_get_cost_per_unit`` helper DEFAULTS A MISSING KEY TO 0.0, not to the full 

137# input rate. Concretely: ``cost_per_token(model="azure/deepseek-v4-pro", 

138# prompt_tokens=1000, completion_tokens=100, cache_read_input_tokens=900)`` 

139# returned a prompt_cost of $0.000193 (only the 100 uncached tokens billed) — 

140# the other 900 tokens cost nothing. Runs show ~93% cache-hit on this route 

141# (dev standard / test_implementer / manager_watcher — the heaviest-volume 

142# model), so historical ``runs.cost_usd`` for those rows UNDERSTATES real 

143# spend by roughly that fraction, not overstates it. 

144# 

145# No Azure retail meter publishes a separate cached-token rate for this 

146# deployment, so the rate below is an ESTIMATE: LiteLLM's own built-in entry 

147# for the same model on a different host (``fireworks_ai/deepseek-v4-pro``) 

148# publishes cache_read_input_token_cost=$0.145/1M against an input rate of 

149# $1.74/1M — a ~8.33% cache/input ratio. Applying that ratio to our verified 

150# Azure input rate ($1.93/1M) gives the estimate below. Flagged as ESTIMATED 

151# in the registration metadata like the other two rates. 

152_DEEPSEEK_V4_PRO_CACHE_READ_PER_TOKEN = 1.61e-7 # ~$0.161 per 1M (estimated, see above)
